# 数据源管理 演示计划

## 一、演示操作路径

按顺序走，约 2~3 分钟：

**1. 登录并进入模块**
- 浏览器打开平台地址，用 `admin / admin123` 登录
- 点击左侧菜单「数据源管理」

**2. 新增 FILE 数据源**
- 点击左侧表单，数据源类型默认为 FILE
- 填写数据源名称，如「电商文件数据源」
- 观察到：主机、端口、数据库名、用户名、密码字段均不显示
- 填写说明，如 “1月份订单数据” ，点击「创建数据源」
- 观察：右侧列表出现新增记录，状态为 ENABLED

**3. 新增 MYSQL 数据源**
- 数据源类型切换为 MYSQL
- 观察：连接配置字段出现
- 填写：
  - 数据源名称：`电商MySQL数据源`
  - 主机：`127.0.0.1`，端口：`3306`
  - 数据库名：`datalake`，用户名：`root`，密码: `123456`
  - 同连接处理策略：不用选，默认就是 WARN
- 点击「创建数据源」，右侧列表出现新记录
- 口述：MYSQL 数据源配置好后，可在数据接入页直接选库选表导入，不需要手动导出文件

**4. 测试连接**
- 找到刚创建的 MYSQL 数据源，点操作列「测试连接」，这里会失败，因为刚才的账号密码乱输的, 如果输入正确会提示连接成功

**5. 演示重复连接策略（WARN）**
- 再次新增一个 MYSQL 数据源，名称改为「电商MySQL数据源-副本」
- 连接信息与上一条完全相同，策略保持 WARN
- 点击「创建数据源」
- 观察：保存成功，同时弹出警告弹窗，提示检测到与现有数据源使用相同连接配置

**6. 停用 / 启用**
- 找到任意一条数据源，点「停用」
- 弹出确认框，点确认
- 观察：列表中该数据源状态变为 DISABLED
- 再点「启用」，状态恢复为 ENABLED

**7. 编辑数据源**
- 点击某条数据源的「编辑」
- 观察：左侧表单回填该数据源信息，按钮变为「保存修改」
- 修改说明字段内容，点「保存修改」
- 观察：列表对应记录的说明已更新，表单恢复新增状态

**8. 删除**
- 点击「副本」数据源的「删除」
- 观察：删除成功，列表中该记录消失
- 口述：若数据源下已有数据集，删除会被系统拒绝，防止数据集失去来源

## 二、说明稿

**开场**

我演示的是数据源管理模块。数据源是整个数据湖的入口，所有数据集的创建都依赖于数据源配置。

**新增 FILE 数据源**

平台支持两种数据源类型：文件类型 和 MYSQL数据库类型。FILE 类型本质上是一个来源标签，标记"这批数据是通过文件手动上传进来的"，本身不代表真实的文件系统连接。先创建一个 FILE 类型的数据源。

**新增 MYSQL 数据源**

切换到 MYSQL 类型，填入连接信息，这里有一个同连接处理策略的设置，支持 （ALLOW、WARN、REJECT） 三种模式，默认是 WARN——如果检测到与现有数据源使用相同的连接配置，系统会允许保存但弹出提示，防止重复注册同一个数据库。

**测试连接**

数据源创建完成后，可以点「测试连接」，系统会真实尝试建立 数据库 连接，成功会返回数据库类型信息。失败则提示连接失败。

**重复连接策略演示**

再注册一个指向同一数据库的 MYSQL 数据源，填写相同连接信息，可以看到保存成功，但系统弹出提示，告知已有相同连接配置的数据源，建议复用。这就是 WARN 策略的效果。

**停用 / 启用**

数据源支持启用和停用操作，停用后不会删除数据，只是标记为不可用状态，方便后续恢复。

**删除保护**

删除时系统会做保护性校验——如果该数据源下已有数据集，删除请求会被拒绝，避免数据集失去来源引用。

## 三、后端实现说明（应对提问）

**Q：数据源信息存在哪里？**

存在 MySQL 的 `data_source` 表里，字段包括类型、连接信息、状态等。密码明文存储，查询列表时不返回密码字段，编辑时留空则保留原密码。

**Q：测试连接怎么实现的？**

后端取到数据源连接信息后，FILE 类型因为只是来源标签、没有真实连接，直接返回"状态正常"；MYSQL 类型用 `DriverManager.getConnection()` 真实建立 JDBC 连接，连接成功后读取 `DatabaseMetaData` 返回数据库类型信息，随后关闭连接。

**Q：重复连接策略怎么判断的？**

新建或编辑时，后端对所有同类型已有数据源做比对，匹配条件是 host、port、dbName、username 四个字段全部相同（忽略空白）。命中后按策略处理：REJECT 直接抛异常，WARN 允许保存并在返回体中带上提示文字，ALLOW 静默通过。

**Q：删除为什么有时候会失败？**

删除前会查 `data_set` 表，统计该数据源下的数据集数量，大于 0 则拒绝删除并返回错误提示，防止数据集的来源引用悬空。
